Visual Basic之父回憶往事

天天要聞
1 週 ago

Alan Cooper ,「VB之父」「交互設計之父」，榮獲視窗先鋒獎（Microsoft Windows Pioneer）和軟體夢幻獎（Software Visionary），他創建了Cooper交互設計公司，此公司一直致力於創建專為用戶而設計的應用軟體。

作者 | Alan Cooper

譯者 | 風車雲馬，責編 | Elle

出品 | CSDN（ID：CSDNnews）

以下為譯文：

自從我的公司庫柏(Cooper)換了新老闆之後，兩年來我發現自己在庫柏網站上發布的一些個人信息被悄悄刪除了。其中之一是我在1995年寫的一篇文章，它講述了Visual Basic的起源故事。這是一段刻骨銘心的、引人入勝的歷史，所以我想在這裡重新發布，與25年前的原始版本沒有任何變化。

VB有史以來出版的第一本書，名為《Visual Basic操作指南》(現在已經出了第二版，由Waite集團出版社出版)。Mitchell Waite在書的前言中稱我為「Visual Basic之父」，我很喜歡這個稱呼，在我的個人傳記中經常引用它。

在20世紀80年代，我主要編寫軟體，然後賣給軟體公司。1986年，我選擇了微軟Windows平台。它不僅支持所有預期的GUI功能，還有一個獨特的、令人滿意的特性:使用動態鏈接庫(DLL)，您可以動態創建配置工具。當我得知這一消息時，我感覺Windows會有一個遠大的未來。不過我還了解到另一個事實:它的shell程序很糟糕。

在業餘時間裡，我開始寫一個比Windows更好的shell程序。我叫它「Tripod」。微軟最初的shell，叫做MSDOS.EXE，它可以說是Windows發展的絆腳石之一。我試圖通過Tripod實現更容易的使用和配置。但直到1987年底，我與客戶見面時，才突然想到了Tripod的關鍵設計策略。當信息技術部經理向我解釋他需要為不同的用戶群創建和發布各種shell解決方案時，我意識到一個難題:沒有所謂的理想shell。每個用戶都需要特定的shell，根據自己的需要和技能水平進行配置。那一瞬間，我看到了shell設計問題的解決方案: 構建一個shell工具集。在這個工具集中，每個用戶都能夠構建他們所需的shell，以滿足他們獨特的應用程序。我不用告訴用戶理想的shell是什麼，他們可以設計自己的、個性化的理想shell。

一旦確定了設計願景，設計的細節就相對容易了。用戶將通過一個工具面板，創建「表單」並使用工具實例。工具集既包括標準控制項，如列表框和按鈕，還包括特殊的、擴展功能的shell，如自動顯示目錄內容的列表框。

我開始研究這個新版本的Tripod，大概幾個月後初具雛形。其突出的一個特性是，界面完全可以通過拖放操作。用戶可以將兩個控制項(我稱之為gizmo)連接在一起，方法是右鍵單擊並從一個gizmo拖動到另一個gizmo。通過一個可視化的箭頭將一個gizmo事件和另一個gizmo方法邏輯地連接起來。後來，微軟放棄了箭頭，但保留了事件/方法模型。

我開始在這個行業里展示Tripod。矽谷的許多公司都看到了這一點，但很少有人表示出真正的興趣。在1988年3月，我把這個模型展示給比爾·蓋茨，他覺得這很「酷」，而且很有潛力，會對整個產品線產生重大影響。在接下來的幾個月里，我們敲定了一筆交易。很快Tripod這個名字曝光率很高，我們就把它改成了「Ruby」。與此同時，我召集了一批程序員精英來構建更高質量的發布版本。

這個團隊由馬克·默克、加里·克瑞金、邁克·吉爾里和弗蘭克·拉布組成。我們做的第一件事是扔掉舊的原型代碼，重新開始。儘管它已經超過25000行C代碼，還包含了許多有價值的想法。在接下來的18個月里，整個團隊努力工作，功夫不負有心人，最終開發完成了Ruby。我們的產品經過了微軟質量保證團隊的審核，並於1990年初完成提交。

Ruby產品在軟體設計方面體現了許多重要的進步，其中有兩個特別成功。正如我提到的，Windows的動態鏈接功能一直吸引著我，但是擁有這些工具和知道如何使用它們是兩碼事。使用Ruby，我終於發現了動態鏈接的兩種實際用途：首先，它是可安裝的，並且可以動態擴展。其次，gizmos的面板可以動態添加。

Ruby附帶了一種小型語言——用於執行shell程序的十幾條簡單命令。但是，這種語言是作為dll實現的，可以在運行時安裝。從最初的討論開始，我和微軟就有了發展這種語言的想法，甚至想用一種「真正的」語言完全取代它。C語言是最常被提到的候選語言，但最終，微軟將其完全替換為QuickBasic。這種語言與前端的新結合是靜態的、永久性的。

在設計過程的早期，我萌生了gizmo動態化的想法。我設想了這樣一個產品:第三方供應商可以編寫自己的gizmo dll，用戶可以將它們添加到特定領域的產品中，而無需重新編譯。我們定義了一個Ruby介面，可以查詢環境中的可執行文件。如果文件響應正確，Ruby就知道它是另一個協作的gizmo，並請求在工具面板中顯示。這個介面最終被稱為VBX。據我所知，這是第一次在操作系統之外實際使用Windows的動態鏈接功能。它的成功充分證明了我的想法和設計是正確的。

因為微軟擁有Ruby的專有權，所以他們可以隨心所欲地使用。最初的意圖是將它作為一個更強大和更複雜的新shell與Windows 3.0一起發布，就像現在的Windows 95一樣。然而，微軟決定推遲發布Ruby，其shell與當時微軟擁有的OS|2的shell幾乎一模一樣；並通過添加QuickBasic將其從面向所有Windows用戶的shell構造集轉換為面向專業程序員的可視化編程語言。一開始，我對微軟的決定感到非常沮喪，並提出了反對意見。然而，最終產品的強大功能給我留下了深刻的印象，很快我就成為了Visual Basic的熱情支持者。

在我從事Ruby開發的所有時間裡，我與微軟簽訂了嚴格的保密協議，不能告訴任何人我在做什麼。在將近四年的時間裡，我一直保持沉默，儘管我知道Ruby會改變編程領域。VB發布後不久，我的朋友Mitchell Waite打電話給我，問我是不是VB中提到的那個Cooper。我承認是我，他邀請我吃午飯，我們在舊金山見面，我向他講述了我的故事。當我講完時，他驚訝地看著我說:「你成為了Visual Basic之父!」

我有兩個兒子，一個9歲，一個10歲。雖然我在他們成長的過程中起了很大的作用，但他們都有獨立思考的能力。當我看著他們以自己的方式成長，並對自己的生活做出自己的決定時，我明白了做一個父親的真正含義。同時，我驕傲的成為VB之父，就像兒子的父親那樣，看著它一步步成長、成熟，最終創造出一種屬於自己的文化。

1994年5月，在亞特蘭大舉行的Windows世界大會上，比爾·蓋茨頒發了7個Windows先鋒獎，其中一個獎項是為了表彰我成功創建了Visual Basic。我感到非常的榮幸！
