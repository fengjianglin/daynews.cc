一個字元串中到底能有多少個字元？我竟然算錯了

2019-12-15 463條評論 1,660次閱讀 203人點贊

依照Java的文檔， Java中的字元內部是以UTF-16編碼方式表示的，最小值是 \\u0000 (0),最大值是\\uffff(65535)， 也就是一個字元以2個位元組來表示，難道Java最多只能表示 65535個字元？

  • 普通的中文字：字元串的長度是2，每個中文字按UTF-8編碼是三個位元組，字元數組的長度看起來也沒問題
  • emojis字元：我們設置了兩個emojis字元，男女頭像。結果字元串的長度是4, UTF-8編碼8個位元組，字元數組的長度是4
  • 生僻的中文字：我們設置了兩個中文字，其中一個是生僻的中文字。結果字元串的長度是3， UTF-8編碼7個位元組，字元數組的長度是3

看起來字元串的字元數和我們預期的有點不一樣，我們的字元串只有兩個unicode字元, 可是輸出結果有時候是2，有時候是3， 有時候是4，為什麼呢？

這還得從Java的歷史說起。

Java最初設計的Charactor用兩個位元組來表示unicode字元，這沒有問題， 因為最初unicode中的字元還比較少， Java 1.1之前採用Unicode version 1.1.5, JDK 1.1中支持Unicode 2.0, JDK 1.1.7支持Unicode 2.1, Java SE 1.4 支持 Unicode 3.0, Java SE 5.0開始支持Unicode 4.0。

直到Unicode 3.0, Java用兩個位元組來表示unicode字元還沒有問題，因為Unicode 3.0最多49,259個字元， 兩個位元組可以表示65,535個字元，還足夠容的下所有的uicode3.0字元。

但是Unicode 4.0(事實上自Unicode 3.1), 字符集進行很大的擴充，已經達到了96,447個字元，Unicode 11.0已經包含137,374個字元。

在Unicode中，為每一個字元對應一個編碼點(一個整數)，用 U+緊跟著十六進位數表示。所有字元按照使用上的頻繁度劃分為 17 個平面（編號為 0-16），即基本的多語言平面和增補平面。基本的多語言平面（英文為 Basic Multilingual Plane，簡稱 BMP）又稱平面 0，收集了使用最廣泛的字元。

這樣一來，Java的Charactor的兩個位元組的設計，已經不足以容納所有的Unicode 4的字元， 所以可能需要4個位元組才能表示擴展字元，所以現在的Charactor代表的已經不再是一個字元 (代碼點 code point), 而是一個代碼單元(code unit)。

  • Code Point：代碼點，一個字元的數字表示。一個字符集一般可以用一張或多張由多個行和多個列所構成的二維表來表示。二維表中行與列交叉的點稱之為代碼點，每個碼點分配一個唯一的編號數字，稱之為碼點值或碼點編號，除開某些特殊區域(比如代理區、專用區)的非字元代碼點和保留代碼點，每個代碼點唯一對應於一個字元。從U+0000 到 U+10FFFF。
  • Code Unit：代碼單元，是指一個已編碼的文本中具有最短的比特組合的單元。對於 UTF-8 來說，代碼單元是 8 比特長；對於 UTF-16 來說，代碼單元是 16 比特長。換一種說法就是 UTF-8 的是以一個位元組為最小單位的，UTF-16 是以兩個位元組為最小單位的。

Java的字元在內部以UTF-16編碼方式來表示，String.length返回的是Code Unit的長度，而不再是Unicode中字元的長度。對於傳統的BMP平面的代碼點，String.length和我們傳統理解的字元的數量是一致的，對於擴展的字元，String.length可能是我們理解的字元長度的兩倍。

有可能你會問， 對於一個UTF-16編碼的擴展字元，它以4個位元組來表示，那麼前兩個位元組會不會和BMP平面衝突，導致程序不知道它是擴展字元還是BMP平面的字元？

其實是不會的， 幸運的是， 在BMP平面中， U+D800到U+DFFF之間的碼位是永久保留不映射到Unicode字元，UTF-16就利用保留下來的0xD800-0xDFFF區塊的碼位來對輔助平面的字元的碼位進行編碼。

UTF-16編碼中，輔助平面中的碼位從U+10000到U+10FFFF，共計FFFFF個,需要20位來表示。第一個整數（兩個位元組，稱為前導代理）要容納上述20位的前10位，第二個整數（稱為後尾代理）容納上述20位的後10位。前導代理的值的範圍是0xD800到0xDBFF,後尾代理的0xDC00~0xDFFF。

可以看到前導代理和後尾代理的範圍都落在了BMP平面中不用來映射的碼位，所以不會產生衝突，而且前導代理和後尾代理也沒有重合。這樣我們得到兩個位元組的，就可以直接判斷它是否是BMP平面的字元，還是擴展字元中的前導代理還是後尾代碼。

國外的有些用戶用emojis字元做自己的昵稱，導致有些系統不能正確的顯示出來，這是因為這些系統粗暴的使用Charactor來表示，在顯示的時候截斷的時候有時候可能不是在正確的代碼點上進行截斷。

我們在進行字元串截取的時候,比如String.substring有可能會踩到一些坑，尤其經常使用的emojis字元。

自 Java 1.5 java.lang.String就提供了Code Point方法， 用來獲取完整的Unicode字元和Unicode字元數量:

publicintcodePointAt(intindex)
publicintcodePointBefore(intindex)
publicintcodePointCount(intbeginIndex,intendIndex)

注意這些方法中的index使用的是code unit值。
