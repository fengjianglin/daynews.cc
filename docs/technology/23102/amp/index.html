阿里的OceanBase資料庫世界第一，底層原來是用了Paxos協議

天天要聞
2019-12-18

前段時間相信都被阿里的OceanBase資料庫刷屏了，它擊敗世界頭號資料庫廠商Oracle，登頂全球第一。先不看新聞內容，光是看標題就足以讓人耳目一新了。又是「擊敗」，又是「第一」，又是「打破世界紀錄」。即使是IT行業的門外漢，相信也會對這個消息感到振奮。但是你知道其底層其實使用的Paxos協議嗎？如果你不知道也沒關係，這篇文章主要就是讓你理解Paxos協議到底是個什麼東西？

一、從分散式理論說起

我們知道分散式架構中有一個基礎的CAP理論，也就是我們的系統最多只能滿足數據一致性(Consistency)、可用性(Availability)和網路分區容忍(Partition Tolerance)三個特性中的兩個。

在互聯網中的大多數場景，其實都是犧牲了強一致性來換取系統的可用性。舉個例子，就好比是在網上買衣服，我們明明看到還有一件，但是付錢的時候卻提示沒有了，這就是犧牲了強一致性，但換來的就是我們的系統可用。

但是這並不是說我們的強一致性沒有什麼用，系統往往只需要保證「最終一致性」，只要這個最終時間是在用戶可以接受的範圍內即可。就好比剛剛那個例子，付款的時候最終會顯示衣服已經沒有了。

為了實現最終一致性，有幾個方案，比如有名的2pc協議和3pc協議，但是都有著各種各樣的問題，這時候一個非常牛的人出來了，名字是萊斯利·蘭伯特，提出了一個非常強大的分散式一致性演算法Paxos。也就是阿里所使用的。

二、故事敘述

我們想要理解Paxos協議，最好的方式就是通過例子，因此我也舉了一個非常簡單的例子。

從前有座山，山裡有座廟，廟裡有一群和尚。廟裡的所有事都有一群特殊的和尚決定，我們叫這些特殊的和尚為長老吧。廟裡面有什麼重大的決議都是大家商量著來，因此大家可以提出一個提議（Proposal），但是提議可能有很多，比如一號提議，二號提議等等，於是給這些提議加了一個編號。每個提議都需要超過一半的長老們同意才能生效。

如果一個提議沒有通過，那就再拿出一個提議，這個大家應該能理解，每一個長老，只會同意大於當前編號的提議。現在問題來了，廟裡面的香火錢考慮是否修建新的房子。針對這件事大家議論紛紛。但是最終要有一個確定性方案。這就是Paxos協議的大致流程，大概意思就是少數服從多數的一致性協議。

它分為兩個階段：

第一階段（預提案）：

（1）寺廟提議者先選擇一個提案編號N，並且將該編號的預提交請求Prepare(N)發送給寺廟的長老們。

（2）長老們收到預提案請求Prepare(N)，比較存儲的最大提案編號PN與N的關係：如果PN < N，則先令PN = N，並且同意提議者請求，發送當前長老們的最大提案編號PN(等於N)和最後一個批准的數據Value(可能為空)。 否則，拒絕請求，同樣發送發送當前長老們的最大提案編號PN(等於N)和最後一個批准的數據Value(可能為空)。這個部分的意思就好比是逐個表決，及時更新贊同最多的方案。

第二階段（提案）：

（1）當預提案Prepare(N)被大多數長老同意時（PN=N），則做出以下提案：

如果所有接收到的PrepareResponse(PN, Value)中Value全部為空（長老們還未批准任何提案），則自己制定一個該值的取值Vn，並向acceptor集合發送提案：Proposal(N, Vn) 。

如果接收到的PrepareResponse(PN, Value)集合中存在不為空的Value，則選取Value不為空的PrepareResponse(PN, ValueNotNull)集合中PN最大的值(PNmax, ValueNotNull)，並將其作為提案發送給長老：Proposal(PNmax, ValueNotNull)

（2）如果一個長老接收到一個提案編號為N的提案Proposal(N, Vn)則：

如果PN為空或者PN < N，則先令PN = N，Value = Vn，並且通過該條提案，返回:ProposalResponse(PN, Value) 。如果PN == N並且Value為空（尚未批准提案號為PN的任何提案），則先令Value = Vn，並且通過該條提案，返回:ProposalResponse(PN, Value) 。如果PN > N則拒絕該提案，回:ProposalResponse(PN, Value)

上面這張圖就是paxos協議的演示，到這一步不知道我們心中會有疑問嘛，也就是如果只有倆長老，每次的提案每個人都只支持一個，這樣會無求無盡。僅此我們還有必要對這種情況進行一個分析。

三、活鎖和選主

當某一proposer提交的proposal被拒絕時，可能是因為acceptor 承諾返回了更大編號的proposal，因此proposer提高編號繼續提交。 如果2個proposer都發現自己的編號過低轉而提出更高編號的proposal，會導致死循環，這種情況也稱為活鎖。

為了保證流程的執行，我們必須選出一個主proposer，作為提案的唯一人選。如果主proposer能和大數派的集合進行通信，並且使用了一個比所有已經批准的提案號都大的編號，那麼它就能成功產生被接受的proposal。批准拒絕已有的提案並且批准更高的提案號，主proposer最終會選擇到一個足夠大的提案號，最終使用提案被選定，從而達到數據一致性的目的。
