汽車E/E電子架構的進化和改革

天天要聞
14 小時 ago

當今汽車上多達數以百計的ECU（電子控制單元），MCU（微控制處理器單元）及其上面運行著的大量的嵌入式軟體代碼，以及複雜的CAN、FlexRay等整車通訊網路註定了汽車不同於其他的IOT設備或智能手機。汽車上的電子電氣架構一直在朝著為智能化和體驗服務的方向在演化和迭代，只是這個過程相比消費電子行業需要更長的時間。

博世2017年在德國的一個汽車行業會議上分享了其在整車電子電氣架構方面的戰略圖。整車電子電氣架構的發展被分為了6個階段：模塊化階段（一個功能一個ECU硬體），功能集成階段，中央域控制器階段，跨域融合階段，車載中央電腦和區域控制器，車載雲計算階段。

目前汽車電子架構尚處於功能集成階段，而新興的造車企業（如Telsa）及傳統車企重點投入的一些未來車型的研發則有陸續採用更先進的整車電子電氣架構，比如智能駕駛域控制器。

大眾汽車也宣布了計劃通過將其現有汽車搭載來自200個不同供應商的70個ECU「減少到三台中央車載電腦」來減少整車軟體的複雜性，從而為智能化控制提供基礎。

Vector也在在ATZ電子雜誌發表了關於汽車電子電氣架構發展趨勢的預測以及軟體和功能安全方面特點的介紹。汽車電子電氣架構發展的三個階段：以控制器為中心的階段、域控制器階段、中央計算機階段。

歐盟發布電子/電氣架構的特點：集中式CAN網路。

大陸電子發布的未來E/E架構概念：分散式E/E系統架構。

從上面幾個Tier1或者行業巨頭的布局規劃分析，汽車E/E架構正在面臨升級或者說改革。未來汽車電子電器架構的發展和車輛網V2X、IT應用、智能化密不可分。

信息技術和消費電子技術的快速發展和技術積累將會向汽車領域傾斜。這些技術的應用和汽車工業的接收需要時間來適應。這就帶來了汽車電子架構的迅速變化。我們不難看到，能源效率正在推動傳統動力系統向高壓混合動力和電動發動機發展。自動駕駛對分散於功能隔離的控制單元的多感測器提出了融合的要求。網路連接和信息娛樂已經把汽車變成了一個分散式的IT系統：可以訪問雲，OTA升級，高速接入地圖服務、多媒體內容、與其他交通工具和周邊基礎設施交互。

人工智慧和數字化雲服務正在創造新的發展機會，汽車將成為網路世界的一部分，未來汽車將不僅僅是一個出行工具，在為用戶提供信息和娛樂的新的可能性的同時，還將提高舒適性和輔助功能的自動化程度，向自動駕駛汽車的方向發展。

這些發展創新需要電子半導體、軟體和通信能力的更快增長。物理結構和功能電子電氣架構都將是適應日益複雜的系統的關鍵。軟體質量、遠程軟體更新、不同廠商軟體的集成、內部軟體開發和安全性，以及更高的數據速率、對各種數據格式的支持、網關概念、故障操作支持和能源效率，都將成為關鍵的成功因素。

特斯拉的一篇報道中提到，來自數字世界的玩家（例如，谷歌、蘋果等）正在向汽車用戶展示可能適用於汽車領域的機會和解決方案。

這種解決方案是面向服務的架構（SOA），包括操作系統（例如，蘋果的iOS）。這種方法允許開發人員創建高效的新功能（應用程序），這些功能可以很容易地集成到設備的整個生態系統中（例如，iPhone）。這種新功能的無縫集成可以幫助為每個用戶提供個性化。此外，遠程更新支持功能優化、質量改進和靈活的生命周期管理。其他的機會來自已經在信息技術中使用的高性能處理器，以及清晰的設計模式，如層次結構和可伸縮性。

高性能處理器使能更高程度的集成

高性能消費電子處理器正在汽車電子架構中使能新的集成平台。這些處理器的使用為汽車領域引入了強大的操作系統——例如，Linux(參見圖1)。

圖1，消費電子領域的高性能處理器將為汽車電子架構使能新的集成平台

GENIVI聯盟正在建立一個軟體架構標準，包括一個操作系統。在不久的將來，其他應用（例如自動駕駛）的ECU將需要這樣的軟體架構。此外，AUTOSAR標準將得到增強，使用POSIX標準創建AUTOSAR自適應平台（AUTOSAR Adaptive Platform）。這些新技術將使得我們能夠用新的方式去設計汽車電子架構。

無縫的分層架構

到目前為止，E/E架構的設計大多採用漸進的方式，重點放在局部解決方案上。這意味著四件事:

1. 功能的劃分集中在計算能力可用的地方

2. 項目特定的和異構的開發方法在各個E/E開發領域中是常見的

3. 所有ECU都遵循OEM的通用系統需求，這常常導致ECU的過度設計

4. 由於職責和目標嚴格分離，ECU的發展集中在ECU的本地優化。

為了解決這些問題，寶馬為下一代汽車的設計創建了一個分層的E/E架構（見圖2）。具有以下幾個優點：

1. ECU分類是基於需求的

2. 適用於每類ECU的統一開發方法替代了本地開發方法

3. 每類ECU都有特定的系統需求

4. 系統級優化是主要關注點，由系統架構師驅動

圖2，強大的集成平台為汽車領域提供了無縫的分層的電子電氣架構

在這個體系結構中，中央計算平台（圖2的頂層，第1類）劃分主要的軟體功能，這些功能主要在內部開發。這些平台提供高性能，並滿足最高的安全要求。集成ECU（第2類）填充了中央計算平台和普通ECU（第3類）之間的差距——例如，部署需要直接訪問感測器或執行器的時間關鍵功能。對於簡單和非特定於OEM的功能，可以接受普通ECU和感測器和執行器（第4類）。理想情況下，這些ECU、感測器和執行器基於常見的OEM或者一級供應商的零件。

一個可伸縮的架構

當今的網路架構的開發，部分是以增加變種和複雜性為代價的。設計模式通常基於發送方和接收方之間的通信，這常常導致了高依賴性和有限的可擴展性。

對於下一代網路架構，中央通信伺服器（CCS）將支持封裝，以便在本地處理可擴展性。這些架構將具有以下主要特性（參見圖3）：

1. 中央信息伺服器和代理將處理所有網路架構信息

2. E/E架構將被封裝，以較小變化支持從基礎版到全功能版本的所有車輛

3. 網路架構通信將是結構化和層次化的

4. 信息在諸如LIN、CAN、FlexRay、乙太網以及發展中的V2V無線協議之間的轉換是流動的物理節點、信息和服務等級將是獨立的，包括防火牆功能

圖3，中央通信伺服器將支持可伸縮的E/E架構

在系統級別，CCS將為物理拓撲和邏輯拓撲提供優化的可能性。例如，在物理級別上，可以用CCS的高性能路由引擎替換子網關。對於邏輯級別的優化，CCS可以解耦發送方和接收方之間的依賴關係，並為逐步引入SOA打下基礎。

面向服務的架構

汽車功能需要與IT和消費電子行業的功能具有相同的創新水平。由於人們對汽車的依賴性越來越大，汽車的功能也變得越來越複雜。一個例子就是從簡單的巡航控制，到自適應巡航控制，再到自動駕駛的演變。今天，系統的功能視圖是基於信號的。功能的劃分受到遺留系統、ECU資源和開發系統的組織架構的顯著影響。只關注於ECU級別不足以掌握複雜性。

因此，我們在BMW為下一代E/E架構引入了基於SOA的方案（見圖4）。SOA方案為整個系統提供了重要的服務抽象。嚴謹的封裝和分層結構支持使用敏捷開發方法和針對介面進行測試，並降低了系統的複雜性。將大大簡化軟體組件在車輛更新換代時的重用。

圖4，面向服務的架構將幫助BMW掌握日益複雜的E/E功能網路

車內和雲端架構

車內E/E架構和雲端架構越來越接近。越來越多的汽車功能與雲端交換數據或部分功能運行在雲端。因此，功能分離、編碼和防火牆等方面變得越來越重要。這些安全模式已經存在於信息技術和消費電子行業，並且可以轉移到汽車領域。

車內網路和後端之間的連接非常重要。主要方面有：

1. LTE、Wi-Fi和未來5G等通信標準

2. 提供優質服務

3. 足夠的帶寬及可用性

雲端為各種應用程序提供了基礎。功能可以在那裡執行（例如，自動駕駛或多媒體）。可以通過雲端為司機提供移動服務。可以對從汽車傳輸到雲端的信息進行分析。

車內和雲端架構的無縫結合，以及前面提到的構建塊（CCS、SOA和分層的E/E體系結構），將成為使我們的E/E基礎設施能夠處理即將到來的創新的重要一步。圖5顯示了可以在ECU或雲端執行的功能的總體系統結構和軟體設計。

圖5，從車內和雲端架構的總體角度來看，安全性和隱私等機制必須同時處理這兩種架構，才能提供無縫的功能。AF =自動駕駛功能，CD =人群數據

車內和雲端架構的無縫設計和文檔是關鍵的成功因素。這包括數據處理、遠程更新和為能夠在ECU或雲端執行的功能的軟體設計。我們將在下一節中討論這些方面。

流程、方法和工具的改變

目前，整個E/E開發過程使用了許多不同的方法和工具。通常來說，軟體項目中的每個部門都有自己的設計原則、工作流和工具。將來，IT行業設計E/E架構的最先進的方法和工具將消除當前異構的情況，創建一個更加同構的開發環境。圖6顯示了當前和未來的狀態。這一技術進步將提供：

1. 從客戶交互到軟體架構的無縫需求過程

2. 基於SOA的E/E架構的無縫、完整建模

3. 使用SOA設計原則封裝分散式開發的內容

4. 啟動敏捷開發流程、聯合Scrum團隊和共享代碼存儲庫

5. 持續集成和早期使用虛擬集成方法

汽車電子的發展將從信息技術和消費電子標準中受益，但必須重新發明許多技術，以滿足嚴格的需求。汽車電子系統還將與傳統的軟體系統有所不同。對安全性、性能、可用性和信息安全的需求導致了對熟練程度和質量的最高期望，而這在消費類電子產品中並不被認為是必要的。聰明的採用和適應信息技術和消費電子方法和技術將帶來許多機會，同時避免削弱傳統軟體系統的陷阱。
