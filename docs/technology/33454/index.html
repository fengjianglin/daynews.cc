網易雲音樂前端模塊動態下發系統

2019-12-23 760條評論 1,911次閱讀 894人點贊

作者：康東揚

鏈接：https://zhuanlan.zhihu.com/p/91386560

本文是《濱江前端技術沙龍》的分享主題《網易雲音樂前端模塊動態下發系統》，主要介紹了團隊的一個內部系統，這是我們在實際生產中遇到痛點時的一個解決方案。

當前的前端代碼部署流程

前端工程的開發流程大概如下圖：

  • 本地進行業務開發、打包構建配置（webpack）；
  • 線上構建機進行代碼打包構建；
  • 部署系統將新的代碼部署到線上伺服器；

    • 流程比較中規中矩，和現在的大部分團隊應該是同一個模式，前端代碼經過重新構建打包後，文件名中的 hash 發生了變化，從而讓客戶端不受緩存影響，可以獲取到最新的代碼。

    尋找痛點

    前端是一種技術問題較少、工程問題較多的軟體開發領域。

    單從一個應用的迭代維護來看，上面的流程並沒有什麼不妥，但是在雲音樂，除了主站應用外，還有各業務線應用和各種 H5 應用。這些全部加起來可能有上百個。

    現在，我們做個簡單的數學：假設一次上線從審批到構建打包再到部署上線所需要花費的最短時間為 30 分鐘，那 100 個應用都上線所要花費的時間：

    100 * 30 = 3000 分鐘 = 50 小時 = 2.08 天

    然而其中很多時候並不值得花這麼多時間來上線，例如：更改配置中心的某個配置、針對 Web APM 的 SDK 的升級、修改幫助手冊文案等。

    如果有個辦法可以進行快速上線，而不是全部走嚴格的上線流程，那麼日積月累的節省的成本將是一個可觀的收益。

    新模式：直接下發代碼

    面對痛點，我們嘗試了一種新的模式：讓某個腳本擁有動態的特性，開發人員通過界面操作可以對這份腳本的內容進行更改，而從直接更新線上代碼。

    我們看看下面的兩張圖片，它們是利用下發系統接入錯誤上發平台 sentry 的步驟圖：

    • 界面上針對某個應用填寫一些 sentry 平台所需要的表單，這些值用於 sentry 的 SDK 初始化；
    • 平台上點擊發布代碼按鈕直接更新代碼。

    在之後如果有版本、配置的更新，直接在系統上重新配置並下發即可。

    該模式操作起來簡單明了，最重要的是，代碼更新的時間大大縮減，可以縮短至 5 分鐘，甚至實時更新。

    5分鐘是目前我們的一個實踐值，這個時間是自己決定的，具體細節後面解釋。現在我們就來為下發 web 前端代碼這個魔術進行揭密。

    踏出勇敢的一步：腳本 API 化

    瀏覽器在向服務端請求一個腳本的時候，瀏覽器拿到的只是一份文本而已，只是，這時服務端會在 response 裡面告訴瀏覽器，這是個腳本，所以瀏覽器就會把返回結果當作腳本進行編譯執行。

    根據這一點，我們可以在 script 標籤引用一個腳本的時候，直接填寫 API 的 url：

    <script src="/api/script/<This is an appId>"></script>

    然後，在後端的介面路由中定義如下：

    // 這是 koa | eggjs 的寫法router.get('/api/script/:appid', controller.script.get);

    最後，在 controller 中組織腳本內容並返回：

    module.exports = class extends Egg.Controller {    async get() {        const { ctx } = this;        // 獲取到應用的ID        const appid = ctx.params.appid;        // 不要緩存        ctx.set('cache-control', 'max-age=0');        ctx.code = 200;        // 腳本內容自由組織        ctx.body = 'console.log("Hello Code Puzzle");';        // 告訴瀏覽器這是個腳本        ctx.type = 'application/javascript';    }}

    這裡有幾個點需要理解：

    ctx.set(‘cache-control’, ‘max-age=0’)：為了瀏覽器每次訪問都拿到最新的代碼，需要告訴瀏覽器不要對返回結果進行緩存；

    ctx.body = ‘console.log(“Hello Code Puzzle”);’：這個是你組織代碼的地方，實際業務中，代碼不會這麼簡單，後面也會簡單講解下我們的做法；

    ctx.type = ‘application/javascript’：有了這個，瀏覽器才會把返回結果當作是腳本，而不是普通的字元串。

    原理很簡單，實現起來也不難，但是實際生產中的情況往往沒這麼簡單，這裡就有個被我們忽視的問題：穩定性問題。

    上述的方法在把腳本當作一個 API 的同時，也直接將服務暴露在用戶面前，在面對大流量訪問或者惡意攻擊的時候，你的服務就會顯得弱小無助，甚至直接崩潰。

    所以，我們需要一個穩定性方案。

    穩定性方案

    為了不讓服務直面客戶端，我們可以在服務前面設立一道防線，現在常見的做法有：

    • API：接入 Gateway 網關；
    • 靜態資源：放入 CDN。

    這裡雖然我們將腳本 API 化，但是其本質還是返回腳本，訪問是不需要登錄或者許可權的，所以這裡採取 CDN 更合適，架構如下：

    CDN 的兩大特性為緩存和回源，利用回源機制，我們可以在腳本服務面前嵌入 CDN，主流程主要分為 3 大部分：

    客戶端訪問腳本：腳本地址不再是 API，而是 CDN 域名的一個腳本地址，當訪問 CDN 上的腳本時，命中緩存則直接返回給客戶端，如果沒有命中緩存，則觸發回源；

    回源機制：事先配置 CDN 回源到某個服務，我們就可以在這個服務中去組織腳本內容，和上述的 controller 代碼一致；

    腳本內容獲取：根據用戶在平台的操作，組織代碼，並將腳本內容存入 MySQL 資料庫中。為了 CDN 回源時提升讀寫速度，我們會將腳本內容備份在 Redis 中，MySQL 相當於一個 fallback。

    在這套方案中，需要注意的一個點是，客戶端訪問的腳本地址仍然是一個不變的地址，所以返回腳本不能是永久緩存，需要設置一個較短的時間。

    這就是上述所說的 5 分鐘的由來，5 分鐘是一個可接受的實踐值。

    接入了 CDN 後，我們也就直接地享受了 CDN 的好處：

    • 根據緩存和回源特性，攔截住了大部分流量、緩解了源站的壓力；
    • 因為 CDN 服務在全國各地都有節點，可以有效解決跨地域訪問問題，降低訪問延時；
    • 利用 CDN 強大的計算能力，可以攔截惡意攻擊、降低「廣播風暴」的影響。

    管理組織下發的代碼

    界面操作 = 操作 schema

    在界面上操作配置一個模塊的時候，在背後相當於操作一份 schema，如下

    在應用引入動態腳本

    最終所有模塊的 schema 會整合到一個 SDK 中，這個 SDK 就可以根據不同的模塊數據進行特定的邏輯處理，SDK 的代碼結構如下：

    (function(modules) {    modules.forEach((module = {}) => {        const { name, vars } = module;        // 邏輯    });})([    { name: 'sentry', vars: [ ... ] },    { name: 'wapm', vars: [ ... ] }])

    代碼中，所有的應用 SDK 的函數主體都是一樣的，只有傳入的 modules 數據根據用戶操作而發生更改。

    最後在應用中引入這個 SDK 腳本即可：

    <script src="https://cdn.net/appid.js"></script>

    不同的處理類型

    在 SDK 中，我們會根據 vars 中的不同處理類型進行不同的操作，例如：插入腳本、插入 HTML、定義變數等，可以根據不同的業務需求進行定製。

    插入腳本：

    schema 為：

    {    "fn": "script",    "url": "https://music.163.com/hello.js",    "async": true,    "onload": "console.log('script onloaded')"}

    SDK 對應的處理函數為：

    function execScript(params) {    const oElem = document.createElement('script');    oElem.src = params.url;    oElem.async = params.async;    document.body.appendChild(oElem);    if (param.onload) {        oElem.onload = function() {            const oScript = document.createElement('script');            oScript.innerHTML = params.onload;            document.body.appendChild(oScript);        }    }}

    當然，其中還省略了 onload 函數中的一些用戶填寫的變數的替換，這樣就可以在啟動腳本中初始化一些公共技術模塊的 SDK。

    定義變數：

    schema 為：

    {    "fn": "var",    "name": "key",    "value": "2bc5b9f43eaf407fa80f4309082a44eb"}

    SDK 對應的處理函數為：

    function execVar(params) {    variables[params.name] = params.value;}
    插入 HTML
    {    "fn": "html",    "selector": "body",    "content": "<div>container</div>"}

    SDK 對應的處理函數為：

    function execHtml(params) {    const container = document.querySelector(params.selector) || document.body;    const oElem = document.createElement('div');    oElem.innerHTML = params.content;    container.appendChild(oElem);}

    通過對不同的行為進行抽象，我們就可以根據一份 schema 來完成各式各樣的操作，從而達到我們的目標。

    系統優化

    到這裡，我們已經做到了代碼的動態下發，但是針對當前的設計，我們仍有優化空間。

    層級配置

    現在我們已經可以不用每次更改都走發布上線流程了，但是當我們的通用模塊發生更改，例如 sentry 的域名發生變更，那麼對於每個應用我們都需要在下發系統中進行配置操作，100 個應用就有100 次操作，而這些操作都是一模一樣的，是冗餘的。

    針對這種情況，我們增加了層級配置的能力：

    在系統中，增加了項目這個概念，可以把它當作是文件夾，項目下有著許多的應用，當我們需要配置公共配置的時候，只需要在項目級別進行操作，那麼系統就會自動將配置合併到應用中並完成代碼下發操作，合併配置時可以想成是：

    Object.assign({}, projectConfig, appConfig);

    連接部署系統，整合公技模塊

    即使有了這套下發體系，但是我們在使用不同的公技平台的時候仍然存在一些不方便，在雲音樂公共技術平台主要有：

    • sentry：錯誤上發平台
    • wapm：前端監控平台
    • 鷹眼：代碼檢測平台
    • 前端的部署系統
    • 下發系統

    每種公技平台都是相互獨立的，所以在使用上也是有著不同的方法，這也就導致了開發者在創建一個新應用的時候會出現：

    • 在部署系統創建一個應用
    • 在 sentry 平台創建一個應用，獲取密鑰
    • 在 wamp 平台創建一個應用，獲取密鑰
    • 在下發系統創建一個應用並配置 sentry、wapm 等模塊

    對於這種繁瑣、重複的操作我們經過抱持著零容忍的態度，而在雲音樂，部署系統可以說是一切應用的集合點，所以我們可以利用它來做一些事情：

    • 在用戶創建應用的時候自動在各個公技平台創建相同的應用；
    • 部署系統將各公技平台的初始化配置發送給下發系統；
    • 部署系統自動注入動態腳本。

    總結

    前端模塊動態下發系統是雲音樂應用數量達到一定程度後尋求的一種提升效率的解決方案，在未來可能會往開源方向前進，如果你有什麼好的點子，也歡迎一起共建。

    我自己是一名從事了多年開發的前端老程序員。我花了三個月整理了一份最適合今年學習的前端學習乾貨，從最基礎到高級的各種框架都有整送給每一位前端小夥伴，想要獲取的可以關注我的頭條號並在後台私信我：前端，即可免費獲取。
