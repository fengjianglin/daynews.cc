導讀

很多公司都在用阿里巴巴的java開發規範，本文會簡要談一下規範的必要性，然後介紹兩個底層原理。最後，手把手教你怎麼擴展阿里巴巴Java開發規範插件：P3C，添加你的自定義規則。

本心法注重內功修養，普通人修鍊需要三個月，而任督二脈打通之高手，三個小時即可煉成。

任督二脈打通條件如下：

1. 熟悉maven

2. 熟悉gradle

3. 熟悉pmd

4. 經常抓網站(熟悉xpath)

5. 會寫java代碼

一、為什麼要定製規範

代碼是誰維護？是gitlab？還是jdk？說到底，還是程序猿這些人類來維護！

代碼首先是給人看的，其次才是給機器。沒人能讀懂，也沒人能維護的代碼那就是個渣 (￣_,￣ )

那麼，問題來了，如何編寫出任何人都看到懂的代碼？答：制——定——規——范！

然後，問題又來了，雖然每個公司都有自己的編碼規範，但往往出於趕進度、怠惰、個人心情&水平&習慣等原因，加之沒有review，最後的代碼就讓人望而生畏。

規範有了沒人遵守，怎麼強制執行呢？ 答：交——給——工——具！(哎~看這鍋甩的弧線像不像研發的淚?)

本文通過介紹java靜態代碼檢查工具PMD、阿里巴巴p3c開源項目，讓你學會怎麼瀟洒的制定自己的代碼規範，解放你的雙手，以及滔滔不絕的說教。

二、基本原理

古人云，君子生非異也,善假於物也。意思是大佬都是會借力的，可四兩撥千斤。現在就介紹下本期的千斤頂。

1、PMD靜態代碼掃描

1.1 概述

PMD(Project Manager Design)是一種開源分析Java代碼錯誤的工具，它通過靜態分析獲知代碼錯誤。即在不運行Java程序的情況下報告錯誤。其本身附帶了許多可以直接使用的規則，利用這些規則可以找出Java源程序的許多問題。

例如：

潛在的bug：空的if/try/catch/finally/switch語句 ；

未使用的代碼：未使用的局部變數、參數、私有方法等 ；

可選的代碼：String/StringBuffer的濫用 ；

複雜的表達式：不必須的if語句、可以使用while循環完成的for循環；

重複的代碼：糊代碼==糊bug ；

循環體創建新對象：創建對象一時爽，一直創建一直爽 ；

資源關閉：Connect，Result，Statement等使用之後確保關閉 。

此外，用戶還可以自己定義規則，檢查Java代碼是否符合某些特定的編碼規範。例如，你可以編寫一個規則，要求PMD找出所有創建Thread和Socket對象的操作①。

1.2 工作原理

PMD的核心是JavaCC解析器生成器。其結合運用JavaCC和EBNF（擴展巴科斯-諾爾範式，Extended Backus-Naur Formal）語法，再加上JJTree，把Java源代碼解析成抽象語法樹（AST，Abstract Syntax Tree）

從根本上看，Java源代碼只是一些普通的文本。不過，為了讓解析器承認 這些普通的文本是合法的Java代碼，它們必須符合某種特定的結構要求。

這種結構可以用一種稱為EBNF的句法元語言表示，通常稱為「語法」 （Grammar）。JavaCC根據語法要求生成解析器，這個解析器就可以用於解析用Java編程語言編寫的程序。

1.3 規則分類

最佳實踐：公認的最佳實踐的規則②；

代碼風格：這些規則強制執行特定的編碼風格③；

設計：幫助您發現設計問題的規則④；

文檔： 這些規則與代碼文檔有關⑤；

容易出錯的規則： 用於檢測被破壞的、非常混亂的或容易發生運行時錯誤的結構的規則⑥；

多線程：這些規則在處理多個執行線程時標記問題⑦；

性能：標記存在性能問題的代碼的規則⑧；

安全：顯示潛在安全缺陷的規則⑨；

2、阿里巴巴Java開發規約插件p3c

p3c主要包含三個部分：

p3c-pmd ，大部分規則實現，基於PMD開發，若想實現自己的規則，可基於此模塊開發（基於maven編譯打包）

idea-plugin ， IntelliJ IDEA插件（基於gradle編譯打包）⑩

4、分析語法樹

可以看到，整棵樹的根節點是CompilationUnit，即編譯單元，代表每個java源文件。

我們首先要找到@RequestMapping的位置，點擊相應的節點，看看游標是否定位到源碼方法聲明位置。

仔細分析ClassOrInterfaceBody，發現存在Annotation節點，這就是我們需要下手的地方了。

注意，由於寫法不同，被分為了NormalAnnotation及SingleMemberAnnotation，為了後續處理XPath可以這麼寫：

1//ClassOrInterfaceBody//Annotation[@AnnotationName='RequestMapping']
